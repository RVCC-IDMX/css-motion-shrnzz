Accessibility & Reflection

Reduced Motion Implementation

I honored users' reduced-motion preference throughout the page using @media (prefers-reduced-motion: reduce). For the micro interaction the lift/scale and shadow are removed and replaced with an immediate color and focus change. For the timing comparison the transitions are disabled so both buttons change state instantly while remaining visually distinct. For the flip card the transform transition is removed so the back is revealed without animation. In every case keyboard focus is preserved with a clear :focus-visible outline so users still receive an accessible, non-animated state cue.

Design Reflection

One of the interfaces I used recently had a navigation bar that expands when clicked and then hides again so it doesn't cover the screen; that felt thoughtful and helpful because it revealed extra controls only when needed and got out of the way quickly. In contrast, I often find autoplaying hero animations or slow, drifting motion to be distracting â€” they draw attention without adding information and can make it hard to focus on the task. The difference usually comes down to purpose and timing: effective motion is purposeful, quick enough to feel responsive (or easing that supports the meaning), and context-aware; ineffective motion is slow, gratuitous, or mismatched to the interaction. From building these demos I learned to keep motion minimal, to choose timing curves that match intent, and to always provide a reduced-motion fallback so the experience works for everyone.
